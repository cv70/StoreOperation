# StoreOperation

连锁餐饮软硬件一体化智能运营系统。

这不是一个泛化的“门店运营平台”概念项目，而是一套围绕真实商业结果设计的软硬件一体化产品方案：通过摄像头、机器人、IoT 设备与 AI 引擎感知门店现场，再由软件平台完成识别、决策、执行和归因闭环，先在 8 周付费试点中证明可归因 ROI，再转为年度合同，最后扩展到区域复制与总部策略管理。

---

## 1. 我们卖什么

我们卖的不是“报表系统”或“AI 看板”，而是帮助连锁餐饮总部持续回答 5 个问题：

1. 今天哪些门店正在漏利润
2. 哪些动作最值得现在执行
3. 谁负责执行，何时完成
4. 执行后到底有没有带来可归因改善
5. 门店现场、人效、流程和设备到底该怎么改

因此，产品的核心形态不是传统 BI，而是三类工作台：

- 店长的“今日利润待办台”
- 督导的“区域指挥台”
- 总部的“策略与 ROI 台”

同时，它不是一个静态工作流系统，而是一套把数据转化为行动的智能系统：

- 预测问题：提前识别缺货、临期、人力错配、活动失控
- 生成动作：自动给出建议，不要求店长先自己分析
- 评估效果：自动判断动作是否有效，而不是只记录完成
- 持续学习：基于采纳、拒绝、执行结果优化规则和模板

它也不是单纯软件系统，而是软硬件协同系统：

- 摄像头负责感知人、货、场的现场事件
- 机器人负责记录和执行部分门店动作
- IoT 设备负责感知温度、电力、设备状态等关键变量
- AI 负责识别、预测、决策和归因
- 软件平台负责协同、复盘和复制

---

## 2. 为什么这个产品有机会商业化

连锁餐饮普遍存在以下结构性问题：

- 缺货、临期报损、活动失控、人效错配每天都在发生，但总部很难实时纠偏
- 一线依赖经验，报表只能解释昨天，不能指导今天
- 总部和门店之间缺少“洞察 -> 动作 -> 责任 -> 结果”的执行闭环
- 很多系统只能展示数据，不能把数据转化为可执行利润动作
- 很多关键问题发生在门店现场，但传统 SaaS 根本看不到现场

这意味着市场真正愿意付费的，不是更多图表，而是更快回收利润。

## 3. 产品定位

## 3.1 一句话定位

帮助 30~300 家门店的连锁餐饮品牌，在 8 周内跑通“现场感知、智能识别、动作生成、执行回写、效果归因”的闭环，并用可归因数据证明 ROI。

## 3.2 北极星指标

- 外部北极星：单店月度可归因利润改善额
- 内部北极星：建议采纳率 x 任务按时完成率

## 3.3 第一阶段只解决什么问题

当前阶段只打透 5 个高价值场景：

1. 智能补货防缺货
2. 临期库存处置
3. 高危告警闭环
4. 高峰排班优化
5. 活动 ROI 控盘

并优先打透 3 个硬件增强场景：

1. 摄像头合规与现场识别
2. 送餐/出餐链路追踪
3. 冷链与关键设备监控

原则很简单：所有能力都必须能指向增量毛利、降本或控险。

## 3.4 智能化体现在哪里

本项目的智能化不是聊天机器人式交互，而是经营决策智能化，主要体现在四层：

1. 预测智能
   - 预测补货需求、临期风险、班次压力、活动收益波动
   - 预测设备故障、冷链风险、送餐拥堵
2. 决策智能
   - 把数据异常转成可执行动作，并按预估收益排序
3. 归因智能
   - 自动关联动作与结果，区分“做了”和“做有效了”
4. 学习智能
   - 根据采纳率、误报率、执行结果持续优化模板和阈值

## 3.5 硬件化体现在哪里

本项目的硬件化不是为了卖设备，而是为了让系统直接感知门店现场。核心设备包括：

1. 智能摄像头
   - 识别员工违规、前厅拥堵、出餐堆积、收银异常
2. 送餐机器人
   - 记录点餐到收餐的完整链路，并优化路径和协同
3. IoT 传感器
   - 监控冷链、温湿度、电力和设备在线状态
4. 边缘计算盒子
   - 承载本地视频解析和低时延事件处理

## 4. 目标客户

## 4.1 核心 ICP

- 连锁餐饮，30~300 家门店
- 以直营或强管控加盟为主
- 总部具备运营负责人和督导体系
- 至少可开放 POS、库存、采购三类数据
- 愿意在试点门店部署基础摄像头或 IoT 设备
- 愿意按周复盘并指定业务 Owner

## 4.2 当前不优先服务的客户

- 10 家以下小连锁
- 没有基本数据系统的品牌
- 完全松散加盟、总部控制力弱的品牌
- 超大集团级客户的复杂定制项目

原因不是这些客户没价值，而是它们不适合作为第一阶段标准化产品的主战场。

## 5. 商业化与实施路径

产品必须按“先试点、再转正、后扩店”的路径经营，而不是一次性卖全套平台。

## 5.1 标准路径

1. 价值诊断
   - 识别客户当前利润损失结构
2. 8 周付费试点
   - 3~5 家门店
   - 冻结 3~5 个必赢场景
   - 每周复盘 ROI
3. 年度转正
   - 按门店软件订阅
   - 附带 AI 场景包和硬件包增购
4. 区域扩店
   - 从试点店复制到区域门店
5. 总部增购
   - 督导效率、供应商治理、加盟治理、新店爬坡等

## 5.2 收费结构

- 试点服务费
- 年度软件订阅费
- AI 场景包费用
- 硬件销售或租赁费
- 运维服务费
- 可选成果激励费

## 5.3 实施原则

- 先验证 ROI，再追求功能完整
- 先打透 3~5 个场景，再考虑扩边界
- 先建立组织闭环，再扩展数据、设备和模型复杂度
- 先复制到区域，再讨论总部级平台化能力

## 6. 产品结构

## 6.1 三个核心工作台

### 今日利润待办台

面向店长，默认展示：

- 今日可回收利润估计
- Top 利润动作
- 风险事件与处理截止时间
- 任务执行状态
- 本周改善趋势

### 区域指挥台

面向督导，默认展示：

- 高风险门店排行
- 逾期任务
- 重点场景改善进度
- 可复制动作模板

### 总部策略台

面向总部运营与管理层，默认展示：

- 区域对比
- 场景 ROI
- 策略模板
- 扩店进度
- 转正与续费健康度

## 6.2 八个底层引擎

- 感知引擎：接入摄像头、机器人、IoT 设备并抽取现场事件
- 预测引擎：预测销量、缺货、临期、人力负载、活动波动和设备风险
- 识别引擎：识别违规操作、拥堵、流程异常、设备异常
- 决策引擎：把预测和规则转成动作建议，并做收益排序
- 任务引擎：把告警和建议自动转成任务
- 归因引擎：把动作和业务结果关联起来
- 模板引擎：支持快餐、茶饮、烘焙等场景模板
- 规则引擎：支撑阈值、升级、分层策略发布

## 8. 产品边界

为了确保这是一款可卖、可交付、可复制的产品，当前阶段明确不做：

- 泛化 BI 平台
- 全自动采购执行闭环
- 复杂选址与动态定价系统
- 面向所有零售业态的通用化方案

说明：

- 我们做的是可交付的软硬件一体化产品，不是从零研发所有硬件本体
- 摄像头、机器人和传感器优先采用标准化接入与生态合作

这不是能力问题，而是产品阶段问题。先打透单一行业、单一利润问题，再逐步扩边界。

---

## 9. 成功标准

这套产品是否成立，不看文档是否完整，而看以下结果是否成立：

- 试点转正率 >= 60%
- 转正客户 6 个月扩店率 >= 150%
- 客户回本周期 <= 6 个月
- 12 个月续费率 >= 85%

如果做不到这四点，这个项目就还不算一个真正成立的商业化产品。
